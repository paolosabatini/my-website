Radiation damage effect in the alignment

The idea is to quantitatively evaluate the effect the radiation damage in the alignment constants. The two studies that were initially thought are:

  • Lorentz-angle drift: this would produce a "rotation" of the IBL depending on eta. Therefore, for a given fixed L3 constant period, this should be compensated by the bowing, ending up in a drifting of the bowing with luminosity from the expected bowing from the operational temperature.
  • Degraded charge collection at large angle: radiation damage reduces the efficiency of charge collection from far in the depth of the sensor. Therefore, you may losing hits at large eta, bringing a "shrinking" of the stave. Since this DoF is just at L3 that is done on larger stats, dedicated alignment only with Ty for pixels for some runs along Run2 could be done to see if there is a systematic effect with lumi.
There are two other ideas to test:
  • Check rotation of the BLayer: Lorentz-angle drift would bring a rotation at first order of the B-layer as 1. reflection of IBL behaviour and 2. lorentz-angle drift itself in BLayer.
  • Align the rad. damaged MC: (cry)

Check the DB entries

It is super easy, and AtlCoolConsole is the command to use. For example, some runs of 2017 for the run 334384 seem to be weird, let's check the bowing.


      $ AtlCoolConsole.py COOLOFL_INDET/CONDBR2
      # wait for the access to the remote offline
      $ >> cd Indet/IBLDist
      $ >> usetag IndetIBLDist_Run2_Legacy_looser_2017_AllRight # the run2 legacy tag for rel.22
      $ >> userunlumi 334384 0 334385 0
This will show the following values:

	[334384,84] - [334384,104) (100) [stave (Int32) : 0], [eta (Int32) : 0], [mag (Float) : -0.00481896847486496], [base (Float) : 0.0], [free (Float) : 0.0]

	...
	[334384,0] - [334384,84) (1400) [stave (Int32) : 13], [eta (Int32) : 0], [mag (Float) : -0.003109687240794301], [base (Float) : 0.0], [free (Float) : 0.0]

So you see, the IoVs of the corresponding constants, for different ranges of LBs. This run has 3 blocks of LBs for which the dynamic constants are delivered: [0,84), [84,104), [104, end-of-run). A value of bowing (indicated as mag) is given for each stave, named as channel (you see different stave IDs). You can select a single stave by selecting the channel (via usechan), in this case (channel) = (stave ID)*100 + 100, but this is not general, each subdetector in each folder has its own encoding of channels.



Copy the content of the DB in a local db file

In this case, we use AtlCoolCopy, to copy the content of the remote DB in a local db file (to manipulate, merge etc..) 


	AtlCoolCopy COOLOFL_INDET/CONDBR2 "sqlite://;schema=mylocaldb.db;dbname=CONDBR2" -create -folder /Indet/IBLDist -tag _SOURCE_TAG_ -outtag _LOCAL_TAG_ -runlumisince _FIRST_RUN_ _FIRST_LB_ -runlumiuntil _LAST_RUN_ _LAST_LB_  -alliov
This copy the content of the folder /Indet/IBLDist from the tag _SOURCE_TAG_ for the IoV [_FIRST_RUN_, _FIRST_LB_] - [_LAST_RUN_, _LAST_LB_] into the local database file mylocaldb.db renaming the tag in _LOCAL_TAG_. In case we need to copy some constants/extend some IoV, one can re-direct the constants into another IoV - warning: be careful with changing IoVs, the "size" must match! - with the commands -newrunlumisince and -newrunlumiuntil.

With these two commands (AtlCoolConsole and AtlCoolCopy) you can do most of the activities in the alignment operations.
  • Investigate the content of local or remote DB, to check whether entries make sense.
  • Manipulate the content of local or remote, by copy-paste values for given IoVs from a source db.
  • Create new tags that can have some IoVs or "static" (going from 0 to infinitiy as LB validity).
A great example are the instructions Javi left for the fix of 2017 bowing constants here:
/afs/cern.ch/user/a/atlidali/w0/calibLoop/checker/CalibrationLoop2016/DynamicIDAlignDBTools/InstructionsUploadFromJavi/InstructionsFixingLegacyForBowingIBL2017.txt

In case of need of creating a static tag, sometimes useful for L3 constants, you can do it with an automated script here:
~/w0/calibLoop/checker/CalibrationLoop2016/StaticTagCreator.

An example of usage is: 

       python IDALignmentConstantsUploadToDB.py --tagInfo 'Static tag for 2017 run 337491. Trying to solve pt+/pt- asymmetry. For testing and CL realignment' --tagFamilyName _pt+-_2017_2ndPart_PixelL3_0 --inputRootFile data17_13TeV.00337491.calibration_IDTracks.idalignsolve.ROOT_CONSTANTS.Step6.Iter0.Block00.rel22_c0._0001.4 --inputDBFile data17_13TeV.00337491.calibration_IDTracks.idalignsolve.ROOT_DB.Step6.Iter0.Block00.rel22_c0._0001.4 --baseRun 337491 --EXEC
What you cannot with these two commands? Push constants into protected tags - see definition in references below. For that, you need to use the next command.



Push constants into a protected (UPD1/UPD4) tags

To do so, AtlCoolMerge command is needed. The syntax is pretty similar to AtlCoolCopy, the only difference is that you need the user/password credential to push - find them hard-coded as passphrase here: /afs/cern.ch/user/a/atlidali/w0/calibLoop/checker/CalibrationLoop2016/DynamicIDAlignDBToolsRun3/CalibrationLoopChecker.py. The syntax is the following: 


	/afs/cern.ch/user/a/atlcond/utils/AtlCoolMerge.py --debug --batch --nobackup --comment "My comment on the tag" --tag MyTagName --passopt='-cti' --folder /Indet/FOLDER /path/to/local/db/file.db CONDBR2 ATLAS_COOLWRITE _USER_ _PSW_
This command is crucial in the CalibrationLoop pushing of the constants. However, this is usually coded into the CL scripts, such as CalibrattionLoopScript.py.



Weakmodepackage @ atlidali


The weakmodepackage has been fully ported to atlidali, and working smoothly from there. At the moment all is stored (or better linked) in the ~/weakmode folder. Here, a small walkthrough of all steps from Zmumu ntuples to maps/plots for iterations. A small description is already in useful_instructions.txt within the folder.

Prepare the batch scripts : given files to run the weakmode analysis over, the command to prepare the "job" scripts is the following: 

      python prepareBatch.py /path/to/zmumu/ntuples/*root
This creates a single folder for each root file, meaning that the analysis is ran over each of the root file separately. In case you want to run over all of them together, you can add MERGE immediately after python prepareBatch.py.

Launch the jobs : once the folders are ready, we can run the jobs. The problem is that currently the code is stoded in /eos/ preventing the batch system to work. Therefore, you can just run the shell script manually, with this snippet: 

      for folder in folder_1 folder_2 .. folder_N; do ./${folder} | tee ${folder}/log ; done
The output is stored, with the folder name, in ~/weakmode. In case in the future the code will be moved again out of /eos/ you can run that with: 

      condor_submit zmumu.sub


Plot the maps : detailed instructions are in this page, the scripts are stored in ~/weakmode/to_scripts/.

Create the maps for the following iteration : the script is stored within ~/weakmode/maps. The base command is: 

      python makeZmumuMapFile.py -i /path/to/input/root/file.root -o /path/to/output/map.root
You can customize the content with some variables at the top of the script:
  • _useVarMin_: it makes use of the VarMin method to fill the dsagitta constrain maps (recommended: True).
  • _injectGlobalSagitta_: if not set to None, it adds the given value (in 1/TeV) onto the values of the dsagitta map (recommedned: None).
  • _fullD0Constr_: it fills the do_constraint map with the value of mean d0 and not the default delta_d0 (recommended: True).
  • _fullZ0Constr_: it fills the do_constraint map with the value of mean z0 and not the default delta_z0 (recommended: False).
  • _doInvertIP_: it flips the sign of the values in the IP (d0 & z0) maps (recommended: True).
  • _doInvertSagitta_: it flips the sign of the values in the dsagitta maps (recommended: False).
